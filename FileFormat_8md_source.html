4 # Overview #
5 
6 [TOC]
7 
8 The %RMF file format (short for Rich Molecular Format) stores
9 hierarchical data about a molecular structure in a file or buffer in
10 memory. This data can include
11 - molecular structures stored hierarchically. These structures need
12 not be atomic resolution.
13 - feature information about parts of the structures, such as how well it
14 fits a particular measurement.
15 - geometric markup such as segments, surfaces, balls, colors which can
16 be used to improve visualization
17 
18 For example, a protein can be stored as a hierarchy where the root is
19 the whole molecule. The root has one node per chain, each chain has
20 one node per residue and each residue one node per atom. Each node in
21 the hierarchy has the appropriate data stored along with it: a chain
22 node has the chain identifier, and a residue node has the type of the
23 residue stored and atom nodes have coordinates, atom type and
24 elements. Bonds between atoms or coarser elements are stored
25 explicitly as dealing with external databases to generate bonds is the
26 source of much of the difficulty of dealing with other formats such as
27 PDB.
28 
29 The file might also include a node for storing the r-value for a FRET
30 measurement between two residues (with links to the residues) as well
31 as extra markers to highlight key parts of the molecule.
32 
33 Multiple conformations on the hierarchy are stored as frames. Each
34 frame has the same hierarchical structure, but some aspects of the
35 data (eg coordinates) can have different values for each frame (or no
36 value for a particle frame if they happen not be be applicable then).
37 
38 A hierarchical storage format was chosen since
39 - most biological molecules have a natural hierarchical structure
40 - it reduces redundancy (eg the residue type is only stored once, as is
41 the residue index)
42 - most software uses a hierarchy of some sort to represent structures at
43 runtime, so less translation is needed
44 - low resolution and multiresolution structures are then more natural
45 as they are just truncations of a full, atomic hierarchy.
46 
47 In addition to structural information, the file can also store
48 information about
49 - bonds
50 - how different parts of the structure relate to different experimental data
51 - different organization schemes on the structure
52 - arbitrary extra data needed by other programs
53 - associated authorship and publication information

68 # The RMF Hierarchy # {#hierarchy}
69 
70 More technically, each node in the %RMF hierarchy has
71 
72 - a type (RMF::NodeType)
73 
74 - a human readable name (RMF::NodeHandle::get_name())
75 
76 - an ID that is unique within the file (RMF::NodeHandle::get_id())
77 
78 - and associated attributes.
79 
80 One accesses nodes in the hierarchy using handles, RMF::NodeHandle and
81 RMF::NodeConstHandle. The root handle can be fetched from the
82 RMF::FileHandle using RMF::FileHandle::get_root_node().
83 
84  # Attributes # {#attribute}
85 
86 Each attribute is identified by a key (eg RMF::IntKey) and is defined by a
87 unique combination of
88 
89 - a \ref decoratorsattributes "category" such as \ref physics "physics",
90 identified by an RMF::Category.
91 
92 - a name string
93 
94 On a per %RMF basis, the data associated with a given key can either
95 have one value for each node which has that attribute, or one value
96 per frame per node with the attribute. The methods in RMF::NodeHandle
97 to get and set the attributes take an optional frame number.
98 
99 The library provides decorators to group together and provide easier
100 manipulation of standard attributes. Examples include RMF::Particle,
101 RMF::decorator::Colored, RMF::decorator::Ball etc. See [Decorators](\ref decoratorsattributes) for
102 more information.
103 
104 The data types that can currently be stored in an RMF file are
105 | Name | Description | C++ type | Python type |
106 |--------:|--------------|:--------:|:-----------:|
107 | RMF::Float | a floating point value | `float` | `float` |
108 | RMF::String | a utf8 string | `std::string` | `string` |
109 | RMF::Int | an 64 bit integer | `int` | `int`|
110 | RMF::Vector3 | three Float values | RMF::Vector3 | RMF.Vector3 |
111 | RMF::Vector4 | four Float values | RMF::Vector4 | RMF.Vector4 |
112 
113 In addition, an arbitrary length list of any of the above can be stored. The type for that
114 is the type for the single data with an \c s on the end, eg \c Floats for a list of \c Float
115 values. These are passed as \c std::vector like lists in \c C++ and \c lists in \c Python.
116 
117 Each data type has associated typedefs such as
118 | Name | Type |Role |
119 |--------:|:-----:|--------------------------------------|
120 |RMF::Float | `float` | the type used to pass a floating point value |
121 |RMF::Floats | std::vector<RMF::Float> | the type used to pass a list of floating point values. It looks like an \c std::vector in \c C++ and a \c list in \c Python |
122 |RMF::FloatKey | RMF::ID<RMF::FloatTraits> | a RMF::Key used to identify a floating point value associated with a node in the RMF hierarchy |
123 |RMF::FloatsKey | std::vector<RMF::FloatKey> | a RMF::Key used to identify a list of floating points value associated with a node in the RMF hierarchy |
124 |RMF::FloatTraits | RMF::FloatTraits | a traits classes to tell HDF5 how to read and write one or more floating point values |
125 |RMF::FloatsTraits | RMF::FloatsTraits | a traits classes to tell HDF5 how to read and write one or more lists of floating point values |

128 # Inheritance of properties # {#inheritance}
129 
130 Certain nodes modify how their children should behave. This
131 modification can be either through inheritance (eg all descendants are
132 assumed to have the property unless they explicitly override it) or
133 composition (the descendant's property is the ancestors composed with
134 theirs). Note that since a given node can be reached through multiple
135 path in the hierarchy, a given view of the file might have to have
136 multiple objects (eg graphics) for a single node.
137 
138 Current examples are - RMF::Colored is inherited. That is, a node that
139 is not an RMF::Colored node, has the color of its closest RMF::Colored
140 ancestor. - The RMF::Particle and RMF::RigidParticle coordinates are
141 transforms of any RMF::ReferenceFrame(s) that occur on the particle
142 itself or its ancestors. That is, a node that is a RMF::Particle or
143 RMF::Ball with an ancestor that is an RMF::ReferenceFrame has global
144 coordinates at the RMF::ReferenceFrame's transformation applied to its
145 coordinates. See the example rigid_bodies.py.

147 # Frames # {#frames}
148 
149 Each RMF file stores one or more frames (conformations). The
150 attributes of a node in a given conformation are the union of
151 conformation-specific attributes as well as static attributes (values
152 that hold for all frames).
153 
154 As with nodes, frames have a hierarchical relationship to one
155 another. This hierarchy supports natural representation of clustering
156 results (eg you have a frame for the cluster center with a child frame
157 for each conformation that is in the cluster). By convention,
158 sequential frames in a simulation should be stored as with the
159 successor frame as a child of the predecessor.
160 
161 Frames also have arbitrary attributes associated with them, like
162 nodes.

165 # Adding custom data to an RMF # {#adding}
166 
167 When adding data to an %RMF file that is just to be used for internal
168 consumption, one should create a new category. For example,
169 [IMP](http://www.integrativemodeling.org) defines an ''imp'' category
170 when arbitrary particle data are stored.
171 
172 If, instead, the data is likely to be a general interest, it probably
173 makes sense to add it to the documentation of this library so that the
174 names used can be standardized.

176 # On disk format # {#on_disk}
177 
178 The RMF library has supported various on-disk formats. Currently 3 output
179 methods are supported: files with suffix `.rmf` and `.rmfz` and buffers
180 in memory.
181 
182 The current format stores the structure in an [Avro Object
183 Container](http://avro.apache.org/docs/1.7.5/spec.html#Object+Container+Files). If
184 the `.rmfz` suffix is used, the contents are compressed. The structure
185 is stored as a series of records, each containing either a frame or
186 static data (there can be multiple static data frames - they are
187 implicitly merged). Upon opening, the file is scanned once; after
188 that, frames can be accessed in a random access fashion. See
189 [Frame.json](Frame.json) for the schema.
190 
191 The format is robust to corruption (all on disk data are safe if
192 garbage data is written or the process is killed).
193 
194 There are several ways that the files can be made more compact
195 (without breaking forwards compatibility of existing files). They can be investigated
196 further if there is sufficient demand.

198 # Benchmarks # {#benchmarks}
199 
200 A quick comparison of the various options (taken from benchmark/benchmark_rmf.cpp).
201 
202 | type | create | write frame | open | traverse | read frame | size |
203 |-----:|-------:|------------:|------:|---------:|-----------:|------|
204 |rmf | 0.11 | 0.05 | 0.2 | 0.03 | 0.04 | 14M |
205 |rmfz | 0.09 | 0.10 | 0.5 | 0.03 | 0.05 | 10M |
206 |buffer| 0.09 | 0.10 | 0.5 | 0.03 | 0.05 | 10M |
207 
208 The operations are:
209 - `create`: create a RMF file with a hierarchy with 45000 particles
210 - `write frame`: save coordinates for those particles
211 - `open`: open the file
212 - `traverse`: traverse the loaded hierarchy, touching atom, residue and chain data
213 - `read frame`: load the coordinates from a frame
214 - `size`: the size of the file. The raw data saved is 11M.
215 
216 Note that the file stayed in RAM for these operations (hence the identical buffer and to disk times).
