4 # Overview #
5 
6 [TOC]
7 
8 The RMF library provides an intermediate level interface to facilitate I/O of
9 RMF data into and out of programs. The primary classes of interest are
10 RMF::RootHandle representing the root of an RMF hierarchy and RMF::NodeHandle
11 representing a node in the hierarchy.
12 
13 The file is automatically closed when the last handle to it is destroyed.
14 
15 # Basics # {#basics}
16 
17 The library defines many classes, some of which are implemented using C++
18 templates. Every class supports output to a `std::ostream` in C++ and
19 conversion to `str` in Python. In addition, every class can be compared
20 to other instances of the same class and can be inserted into hash tables both
21 in C++ and Python. The methods necessary to support these things are
22 omitted for brevity.
23 
24 In addition, there is a typedef for each type for managing lists of the objects.
25 For example, a list of RMF::Category objects is passed using a RMF::Categories type.
26 It looks like a `std::vector` in `C++` and is a `list` in Python.
27 
28 # Associations # {#associations}
29 
30 The RMF wrapper has the concept of an association between nodes in
31 its hierarchy and objects in the program accessing. The methods
32 RMF::FileConstHandle::get_node_handle_from_association(),
33 RMF::NodeConstHandle::set_association() and
34 RMF::NodeConstHandle::get_assocation() can be used to take advantage of
35 this. The idea is that one can store pointers to the programatic
36 data structures corresponding to the nodes and so avoid maintaining
37 ones own lookup table. Any function used in an association must support
38 a `get_uint()` function. An implementation is provided for pointers.
39 
40 # Parallel I/O # {#parallel}
41 
42 If RMF::FileHandle::flush() has been called since the last change, it is safe
43 to read the file from another process. Writing from more than one process is not
44 supported. Nor is reading or writing from more than one thread of the same
45 program.
46 
47 # Invariants # {#invariants}
48 
49 Currently, there is little explicit checking of invariants between attributes
50 in the RMF file. An extensible framework for checking invariants on file
51 close and open will be added.

53 # Decorators and factories # {#rmfdecorators}
54 
55 The nicest way to interact with data stored in nodes is through decorators and
56 factories (see the RMF::decorator). A factory (eg RMF::decorator::ParticleFactory or RMF::decorator::ParticleConstFactory) is
57 constructed from a file (RMF::FileHandle or RMF::FileConstHandle). It can be used
58 to check if given nodes have the requested data at a given frame (eg if the RMF::NodeConstHandle has
59 mass, coordinates and radius and so is an RMF::decorator::Particle) and construct a decorator
60 for that node at that frame. The decorator (eg RMF::decorator::Particle or RMF::decorator::ParticleConst)
61 can then be used to access the attributes and set them (for non-const variants). The
62 accessing is done through appropriately named functions (eg RMF::decorator::Particle::set_radius()).
63 
64 # Language bindings # {#bindings}
65 
66 The RMF library currently supports C++ and Python. The API is
67 written so that [SWIG](http://www.swig.org) can be used to
68 easily generate bindings for most languages. The two main
69 exceptions are C and Fortran. Until the SWIG %C target support is
70 finished, these can be supported by writing a simple %C %API
71 manually, probably a week's work.
